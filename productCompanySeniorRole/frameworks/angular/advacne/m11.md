# APPLICATION STARTUP PERFORMANCE

Startup time depends on:

• bundle size
• number of modules
• number of components
• dependencies

Large Angular apps can easily reach **multiple megabytes of JavaScript**.

To optimize startup performance:

• use lazy loading
• split application into feature modules
• remove unused dependencies

Example lazy route:

```ts
{
 path: 'admin',
 loadChildren: () =>
  import('./admin/admin.module')
   .then(m => m.AdminModule)
}
```

Lazy loading ensures the module loads **only when required**.

# CHANGE DETECTION COST

Angular checks the component tree during every change detection cycle.

Large trees can be expensive.

Example component tree:

```text
AppComponent
   ↓
DashboardComponent
   ↓
WidgetComponent
   ↓
ChartComponent
```

If change detection runs frequently, Angular must check **every component binding**.

This increases CPU usage.

Solution:

```ts
ChangeDetectionStrategy.OnPush
```

==================================================

# ONPUSH STRATEGY (PERFORMANCE BEST PRACTICE)

Example component:

```ts
@Component({
 changeDetection: ChangeDetectionStrategy.OnPush
})
export class UserComponent {}
```

With OnPush, Angular checks the component only when:

• input reference changes
• event occurs inside component
• observable emits new value

This dramatically reduces unnecessary rendering.

# OPTIMIZING LARGE LIST RENDERING

Rendering large lists is a common performance problem.

Example:

```html
<li *ngFor="let user of users">
 {{user.name}}
</li>
```

If the list changes, Angular re-evaluates every element.

Use `trackBy` to optimize rendering.

Example:

```html
<li *ngFor="let user of users; trackBy: trackById">
 {{user.name}}
</li>
```

Component:

```ts
trackById(index: number, user: any) {
 return user.id;
}
```

Angular can now reuse DOM elements.

==================================================

# VIRTUAL SCROLLING

For extremely large lists, rendering thousands of DOM elements is expensive.

Angular provides **virtual scrolling** via Angular CDK.

Example:

```html
<cdk-virtual-scroll-viewport itemSize="50">
 <div *cdkVirtualFor="let item of items">
  {{item.name}}
 </div>
</cdk-virtual-scroll-viewport>
```

Only visible elements are rendered.

This drastically improves performance.

# PURE PIPES VS IMPURE PIPES

Angular pipes transform template data.

Example:

```html
{{price | currency}}
```

Pipes can be:

### Pure pipes

Run only when inputs change.

### Impure pipes

Run every change detection cycle.

Impure pipes can cause **performance issues**.

==================================================

# TEMPLATE PERFORMANCE ANTI-PATTERNS

Bad example:

```html
{{calculateTotal()}}
```

Angular executes this function during **every change detection cycle**.

Better approach:

```ts
total = calculateTotal();
```

Bind the value instead of calling a function.

==================================================

# ASYNC PIPE PERFORMANCE BENEFITS

Example:

```html
<div *ngIf="users$ | async as users">
 {{users.length}}
</div>
```

Advantages:

• automatic subscription management
• fewer memory leaks
• better performance

==================================================

# DETACHING CHANGE DETECTION

For highly dynamic UI components (like charts), change detection can be detached.

Example:

```ts
constructor(private cd: ChangeDetectorRef) {}

ngOnInit() {
 this.cd.detach();
}
```

Manually trigger updates when required:

```ts
this.cd.detectChanges();
```
