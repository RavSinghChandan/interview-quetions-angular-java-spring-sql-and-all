# CORE IDEA: FRONTEND SECURITY RISKS

Web applications face multiple security risks.

Common threats include:

```text id="secz1"
Cross-Site Scripting (XSS)
Cross-Site Request Forgery (CSRF)
Token theft
Session hijacking
Insecure API communication
```

Frontend frameworks like Angular must protect against these threats.

### Mental Anchor

> Angular focuses primarily on preventing **XSS attacks**.

==================================================

# WHAT IS CROSS-SITE SCRIPTING (XSS)

XSS occurs when malicious scripts are injected into web pages.

Example unsafe HTML:

```html id="sec1"
<div>{{userInput}}</div>
```

If `userInput` contains:

```html id="sec2"
<script>alert("Hacked")</script>
```

The script could execute in the browser.

XSS can allow attackers to:

• steal user tokens
• access session cookies
• manipulate the DOM

==================================================

# ANGULAR'S BUILT-IN XSS PROTECTION

Angular automatically **sanitizes template bindings**.

Example:

```html id="sec3"
<div>{{userInput}}</div>
```

Angular treats the value as plain text.

Output becomes:

```html id="sec4"
&lt;script&gt;alert("Hacked")&lt;/script&gt;
```

The script is **not executed**.

### Mental Anchor

> Angular escapes dangerous HTML automatically.

==================================================

# DANGEROUS DOM OPERATIONS

Some operations bypass Angular's automatic protection.

Example:

```html id="sec5"
<div [innerHTML]="htmlContent"></div>
```

Here Angular must sanitize HTML manually.

Angular removes unsafe tags such as:

```text id="sec6"
<script>
<iframe>
<object>
```

==================================================

# DOMSANITIZER

Angular provides a service called **DomSanitizer**.

It allows developers to mark content as trusted.

Example:

```ts id="sec7"
import { DomSanitizer } from '@angular/platform-browser';

constructor(private sanitizer: DomSanitizer) {}

safeHtml = this.sanitizer.bypassSecurityTrustHtml(html);
```

This disables Angular sanitization.

⚠ Use carefully.

### Mental Anchor

> DomSanitizer bypasses Angular security checks.

==================================================

# SECURING APPLICATION ROUTES

Angular applications often contain protected routes.

Example:

```text id="sec8"
/dashboard
/admin
/profile
```

Unauthorized users must not access these pages.

Angular secures routes using **route guards**.

==================================================

# ROUTE GUARDS

Route guards control navigation.

Common guard types:

```text id="sec9"
CanActivate
CanDeactivate
CanLoad
Resolve
```

Most common is **CanActivate**.

==================================================

# AUTHENTICATION GUARD EXAMPLE

Example authentication guard:

```ts id="sec10"
@Injectable({ providedIn: 'root' })
export class AuthGuard implements CanActivate {

 constructor(private authService: AuthService) {}

 canActivate(): boolean {

   return this.authService.isLoggedIn();

 }

}
```

Route configuration:

```ts id="sec11"
{
 path: 'admin',
 component: AdminComponent,
 canActivate: [AuthGuard]
}
```

If `isLoggedIn()` returns false:

Navigation is blocked.

==================================================

# JWT AUTHENTICATION FLOW

Most modern Angular applications use **JWT authentication**.

Flow:

```text id="sec12"
User logs in
      ↓
Backend validates credentials
      ↓
Backend returns JWT token
      ↓
Token stored in browser
      ↓
Interceptor attaches token to requests
      ↓
Backend validates token
```

Example request header:

```text id="sec13"
Authorization: Bearer <JWT_TOKEN>
```

==================================================

# STORING AUTHENTICATION TOKENS

Common storage options:

```text id="sec14"
LocalStorage
SessionStorage
Cookies
```

Security considerations:

• LocalStorage vulnerable to XSS
• Cookies vulnerable to CSRF

Best practice depends on architecture.

==================================================

# HTTP INTERCEPTOR FOR AUTHENTICATION

Authentication tokens are attached using interceptors.

Example:

```ts id="sec15"
@Injectable()
export class AuthInterceptor implements HttpInterceptor {

 intercept(req: HttpRequest<any>, next: HttpHandler) {

   const token = localStorage.getItem("token");

   const authReq = req.clone({
     setHeaders: {
       Authorization: `Bearer ${token}`
     }
   });

   return next.handle(authReq);

 }

}
```

This ensures every API request includes authentication.

==================================================

# PROTECTING AGAINST CSRF

CSRF attacks occur when malicious sites send requests on behalf of users.

Angular supports CSRF protection through tokens.

Example header:

```text id="sec16"
X-XSRF-TOKEN
```

Backend validates this token.

Angular automatically attaches CSRF tokens when configured.

==================================================

# SECURE API COMMUNICATION

Best practices for Angular API communication:

• always use HTTPS
• validate JWT tokens on backend
• avoid exposing sensitive data in frontend

Frontend security should never replace backend security.

==================================================

# ENTERPRISE SECURITY ARCHITECTURE

Example enterprise authentication flow:

```text id="sec17"
User Login
     ↓
AuthService
     ↓
JWT Token Stored
     ↓
AuthInterceptor
     ↓
Secure API Requests
     ↓
Backend Authorization
```

Routes protected with guards.

Sensitive APIs protected with backend authorization.

# COMMON SECURITY MISTAKES (INTERVIEW FILTER)

Many developers:

❌ disable Angular sanitization unnecessarily
❌ store sensitive data in local storage
❌ rely only on frontend security
❌ expose secrets in frontend code

Senior engineers design **defense-in-depth security**.
