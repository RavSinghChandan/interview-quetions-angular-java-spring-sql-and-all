# CORE IDEA: WHAT DEPENDENCY INJECTION SOLVES

Without dependency injection:

Objects create their own dependencies.

Example:

```ts
class OrderComponent {
  orderService = new OrderService();
}
```

Problems:

• Tight coupling
• Hard to test
• Hard to replace implementations
• No lifecycle management

Dependency Injection solves this.

Instead of creating dependencies, they are **provided externally**.

Example:

```ts
constructor(private orderService: OrderService) {}
```

Angular now manages:

• object creation
• dependency wiring
• lifecycle

### Mental Anchor

> Components ask for dependencies.
> Angular decides how to provide them.

==================================================

# THE ANGULAR INJECTOR

Angular DI works through an **Injector**.

The injector is essentially a **service container**.

It keeps track of:

• service instances
• provider configuration
• dependency graph

Example conceptual structure:

```
Injector
 ├── UserService → instance
 ├── AuthService → instance
 └── LoggerService → instance
```

When a component requests:

```ts
constructor(private userService: UserService)
```

Angular:

1. Looks inside the injector
2. Finds `UserService`
3. Returns existing instance (or creates one)

==================================================

# INJECTOR HIERARCHY (VERY IMPORTANT)

Angular DI is **hierarchical**, not global.

Structure:

```
Root Injector
     ↓
Module Injector
     ↓
Component Injector
```

Example hierarchy:

```
AppInjector
   ↓
DashboardModule Injector
   ↓
DashboardComponent Injector
   ↓
UserWidgetComponent Injector
```

Each injector can override providers.

### Mental Anchor

> Angular searches dependencies **up the injector tree**.

# PROVIDER CONFIGURATION

Services must be registered with Angular.

There are several ways.

---

# METHOD 1: providedIn: 'root' (MOST COMMON)

Example:

```ts
@Injectable({
  providedIn: 'root'
})
export class AuthService {}
```

Meaning:

• Service is registered in **root injector**
• Only **one instance exists**

This is essentially a **singleton**.

Benefits:

• tree-shakable
• memory efficient
• globally accessible

==================================================

# METHOD 2: MODULE PROVIDERS

Example:

```ts
@NgModule({
  providers: [AuthService]
})
export class AdminModule {}
```

Service instance exists **per module injector**.

If module is lazy-loaded:

➡ new service instance is created.

This allows **module-scoped state**.

==================================================

# METHOD 3: COMPONENT PROVIDERS

Example:

```ts
@Component({
  providers: [UserService]
})
export class UserComponent {}
```

Each component instance gets its **own service instance**.

Example scenario:

```
UserComponent A → UserService instance A
UserComponent B → UserService instance B
```

Useful for:

• isolated state
• reusable widgets

But dangerous if misused.

==================================================

# DEPENDENCY RESOLUTION ALGORITHM

When Angular needs a dependency:

Example:

```ts
constructor(private authService: AuthService) {}
```

Angular performs the following steps:

Step 1
Look inside **component injector**

Step 2
If not found → check **parent injector**

Step 3
Continue upward until root injector

Step 4
If still not found → throw error

Example error:

```
NullInjectorError: No provider for AuthService
```

### Mental Anchor

> Dependency resolution flows **upward in injector hierarchy**.

==================================================

# TREE-SHAKABLE PROVIDERS

Angular introduced tree-shakable providers for performance.

Example:

```ts
@Injectable({
  providedIn: 'root'
})
```

If a service is never used:

➡ Angular removes it from bundle.

This reduces **bundle size**.

Important for:

• performance
• production optimization

==================================================

# INJECTION TOKENS

Sometimes dependencies are not classes.

Example:

Configuration values.

Angular uses **InjectionToken**.

Example:

```ts
export const API_URL = new InjectionToken<string>('API_URL');
```

Provider:

```ts
{ provide: API_URL, useValue: "https://api.example.com" }
```

Usage:

```ts
constructor(@Inject(API_URL) private apiUrl: string) {}
```

This enables DI for:

• configs
• primitives
• dynamic values

==================================================

# OPTIONAL & MULTI PROVIDERS

Angular DI supports advanced patterns.

Optional injection:

```ts
constructor(@Optional() private logger?: LoggerService) {}
```

If service doesn't exist:

➡ Angular injects `null`.

---

Multi providers:

```ts
providers: [
  { provide: HTTP_INTERCEPTORS, useClass: AuthInterceptor, multi: true }
]
```

Allows **multiple providers for same token**.

Used heavily in:

• interceptors
• plugin systems

==================================================

# LAZY LOADED MODULES & DI

Lazy modules create **child injectors**.

Example routing:

```ts
{
 path: 'admin',
 loadChildren: () => import('./admin/admin.module')
   .then(m => m.AdminModule)
}
```

AdminModule receives its **own injector**.

Meaning:

Services defined in AdminModule are **isolated**.

This is critical for:

• large apps
• microfrontend style architecture

==================================================

# PERFORMANCE INSIGHT

DI has runtime cost.

Angular must:

• resolve dependencies
• construct dependency graph
• create instances

Best practices:

✔ use `providedIn: root` when possible
✔ avoid unnecessary providers
✔ avoid deep injector chains
