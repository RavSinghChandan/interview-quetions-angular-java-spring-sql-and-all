# CORE IDEA: WHY CHANGE DETECTION EXISTS

Angular UI is driven by **data changes**.

Example:

```ts
user.name = "Chandan";
```

The DOM must update automatically.

Angular must answer:

```text
When did data change?
Which components must re-render?
```

The system responsible for this is:

```text
Angular Change Detection Engine
```

### Mental Anchor

> Change detection synchronizes **application state with the DOM**.

==================================================

# THE BASIC CHANGE DETECTION FLOW

Angular change detection works like this:

```text
Event occurs
     ↓
Angular detects async activity
     ↓
Change detection cycle starts
     ↓
Component tree checked
     ↓
Templates re-evaluated
     ↓
DOM updated
```

Example event:

• button click
• HTTP response
• timer
• user input

==================================================

# ROLE OF ZONE.JS

Angular uses **Zone.js** to detect asynchronous operations.

Examples of async tasks:

```text
setTimeout
HTTP requests
DOM events
Promises
```

Zone.js patches these APIs.

Example conceptually:

```text
Original setTimeout
        ↓
Zone.js wrapped setTimeout
        ↓
Angular notified when task completes
```

This allows Angular to know:

```text
"Something happened — run change detection"
```

### Mental Anchor

> Zone.js tells Angular **when something asynchronous happened**.

==================================================

# CHANGE DETECTION TREE

Angular maintains a **component tree**.

Example:

```text
AppComponent
   ↓
DashboardComponent
   ↓
UserListComponent
   ↓
UserCardComponent
```

Angular checks components **top-down**.

```text
AppComponent
   ↓
DashboardComponent
   ↓
UserListComponent
   ↓
UserCardComponent
```

Each component's bindings are evaluated.

Example template:

```html
<h2>{{user.name}}</h2>
```

Angular checks:

```text
previous value vs new value
```

If different → DOM update.

==================================================

# DEFAULT CHANGE DETECTION STRATEGY

By default Angular uses:

```ts
ChangeDetectionStrategy.Default
```

Behavior:

```text
Any async event
      ↓
Entire component tree checked
```

Example:

User clicks button in a small component.

Angular checks:

```text
AppComponent
DashboardComponent
UserListComponent
UserCardComponent
```

Even if most components didn't change.

This guarantees correctness but can affect performance.

==================================================

# ONPUSH CHANGE DETECTION (PERFORMANCE OPTIMIZATION)

Angular allows optimized detection using:

```ts
ChangeDetectionStrategy.OnPush
```

Example:

```ts
@Component({
 changeDetection: ChangeDetectionStrategy.OnPush
})
```

Now Angular only checks the component when:

1️⃣ Input reference changes

Example:

```ts
this.user = {...this.user, name: "Chandan"}
```

2️⃣ Event occurs inside component

3️⃣ Observable emits

4️⃣ `markForCheck()` called manually

### Mental Anchor

> OnPush checks **only when necessary**.

==================================================

# CHANGE DETECTION EXAMPLE

Default strategy:

```text
Button click
      ↓
Angular checks entire component tree
```

OnPush strategy:

```text
Button click
      ↓
Only affected component subtree checked
```

This drastically improves performance in large applications.

==================================================

# MANUAL CHANGE DETECTION

Angular allows manual control through `ChangeDetectorRef`.

Example:

```ts
constructor(private cd: ChangeDetectorRef) {}
```

Manually trigger detection:

```ts
this.cd.detectChanges();
```

Or mark component for checking:

```ts
this.cd.markForCheck();
```

This is useful in:

• performance critical components
• custom async flows

==================================================

# DETACHING CHANGE DETECTION

Angular allows disabling change detection temporarily.

Example:

```ts
this.cd.detach();
```

Angular stops checking the component.

Later re-enable:

```ts
this.cd.reattach();
```

Used for:

• real-time dashboards
• heavy UI rendering

==================================================

# IMMUTABLE DATA PATTERN

OnPush works best with **immutable data structures**.

Bad example:

```ts
this.user.name = "Chandan";
```

Reference unchanged → Angular might skip update.

Good example:

```ts
this.user = {
 ...this.user,
 name: "Chandan"
};
```

Reference changed → Angular detects update.

==================================================

# SIGNALS (MODERN ANGULAR CHANGE DETECTION)

Angular introduced **Signals** for reactive state.

Example:

```ts
import { signal } from '@angular/core';

count = signal(0);
```

Update value:

```ts
this.count.set(5);
```

Angular automatically re-renders components using this signal.

Signals provide:

• predictable reactivity
• reduced change detection cost

# CHANGE DETECTION PERFORMANCE PROBLEMS

Large Angular apps suffer when:

• too many components exist
• expensive template functions run
• default strategy used everywhere

Bad example:

```html
{{calculateTotal()}}
```

Function executes every detection cycle.

Better approach:

```ts
total = calculateTotal();
```
