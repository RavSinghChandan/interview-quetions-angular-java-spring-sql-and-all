# CORE IDEA: DATA FLOW IN ANGULAR

Angular encourages **unidirectional data flow**.

Meaning:

```text
Parent Component
        ↓
   Child Component
```

Data flows **downward through inputs** and **upward through events**.

This ensures:

• predictable UI updates
• easier debugging
• controlled state propagation

### Mental Anchor

> Data flows down, events flow up.

==================================================

# THREE MAIN COMMUNICATION PATTERNS

Angular applications primarily use three patterns:

```text
1. Parent → Child
2. Child → Parent
3. Service-Based Communication
```

Each solves a different problem.

==================================================

# PATTERN 1 — PARENT TO CHILD COMMUNICATION

Parent components pass data using:

```ts
@Input()
```

Example child component:

```ts
@Component({
 selector: 'app-user-card',
 template: `<h3>{{user.name}}</h3>`
})
export class UserCardComponent {

 @Input()
 user!: any;

}
```

Parent component:

```html
<app-user-card [user]="selectedUser"></app-user-card>
```

Angular binds the parent property to the child.

### Mental Anchor

> `@Input()` allows **downward data flow**.

==================================================

# HOW INPUT BINDING WORKS INTERNALLY

Angular evaluates bindings during change detection.

Example:

```html
<app-user-card [user]="selectedUser"></app-user-card>
```

Angular internally checks:

```text
previous reference vs new reference
```

If changed:

Angular calls:

```text
ngOnChanges()
```

inside the child component.

This allows the component to react to input updates.

==================================================

# PATTERN 2 — CHILD TO PARENT COMMUNICATION

Children emit events using:

```ts
@Output()
```

Example child component:

```ts
@Component({
 selector: 'app-user-card',
 template: `<button (click)="selectUser()">Select</button>`
})
export class UserCardComponent {

 @Output()
 userSelected = new EventEmitter<number>();

 selectUser() {
   this.userSelected.emit(101);
 }

}
```

Parent component listens:

```html
<app-user-card
 (userSelected)="onUserSelected($event)">
</app-user-card>
```

Parent method:

```ts
onUserSelected(id: number) {
 console.log(id);
}
```

### Mental Anchor

> Child components **emit events upward**.

==================================================

# PATTERN 3 — SERVICE-BASED COMMUNICATION

When components are **not directly related**, using inputs/outputs becomes difficult.

Example scenario:

```text
NavbarComponent
        ↓
DashboardComponent
        ↓
UserListComponent
        ↓
UserCardComponent
```

Passing events across many layers becomes messy.

Instead, use a **shared service**.

==================================================

# SERVICE COMMUNICATION USING RXJS

Example service:

```ts
@Injectable({
 providedIn: 'root'
})
export class UserService {

 private userSubject = new BehaviorSubject<any>(null);

 user$ = this.userSubject.asObservable();

 setUser(user: any) {
   this.userSubject.next(user);
 }

}
```

Component updating state:

```ts
this.userService.setUser(user);
```

Component listening to state:

```ts
this.userService.user$.subscribe(user => {
 console.log(user);
});
```

This creates **reactive communication between components**.

==================================================

# COMPONENT COMMUNICATION ARCHITECTURE

In large Angular systems, communication flows like this:

```text
Smart Container Component
          ↓
   Presentational Components
          ↓
     User Interaction
          ↓
        Service
          ↓
      State Update
          ↓
        UI Update
```

This architecture improves:

• separation of concerns
• testability
• maintainability

==================================================

# SHARED STATE SERVICES

Services often hold **application state**.

Example:

```ts
@Injectable({ providedIn: 'root' })
export class CartService {

 cartItems$ = new BehaviorSubject<CartItem[]>([]);

 addItem(item: CartItem) {
   const items = this.cartItems$.value;
   this.cartItems$.next([...items, item]);
 }

}
```

Components subscribe to state updates.

```ts
this.cartService.cartItems$.subscribe(items => {
 this.items = items;
});
```

==================================================

# EVENT STREAM ARCHITECTURE

RxJS enables event streams.

Example flow:

```text
User clicks button
       ↓
Component emits event
       ↓
Service receives event
       ↓
State updated
       ↓
Other components react
```

This creates **loosely coupled UI systems**.

==================================================

# COMMON DATA FLOW ANTI-PATTERNS

Senior engineers avoid these mistakes.

---

### 1. Excessive Input/Output Chains

Bad architecture:

```text
Component A
 ↓
Component B
 ↓
Component C
 ↓
Component D
```

Event must travel across many components.

Better approach:

```text
Shared Service
```

---

### 2. Mutating Shared Objects

Bad example:

```ts
this.user.name = "John";
```

Better:

```ts
this.user = {...this.user, name: "John"};
```

Ensures proper change detection.

---

### 3. Business Logic in UI Components

Move logic to services.

Components should mainly:

• render UI
• handle user interaction


This ensures **clean data flow across modules**.

# PERFORMANCE CONSIDERATIONS

Poor communication patterns cause:

• unnecessary change detection
• memory leaks
• circular dependencies

Best practices:

✔ use services for cross-component communication
✔ use `async` pipe instead of manual subscriptions
✔ avoid deeply nested input chains

# 5-MINUTE REVISION SNAPSHOT

> Angular encourages unidirectional data flow
> Parent → child communication uses `@Input()`
> Child → parent communication uses `@Output()`
> Unrelated components communicate through shared services
> RxJS enables reactive communication streams
> Proper architecture prevents tight coupling

==================================================

# INTERVIEW PUNCHLINE (STRONG)

> Angular applications follow a unidirectional data flow where parents pass data to children using inputs and children communicate upward through events. For cross-component communication, shared services combined with RxJS streams enable scalable and loosely coupled UI architecture.
